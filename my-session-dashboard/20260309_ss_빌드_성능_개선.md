## 개요

SQLite DB 기반 증분 빌드로 전환(`session-db.js`) 후, `/ss` 첫 실행이 **1분 이상** 소요되는 성능 저하 발생. 기존 `.build-cache.json` 방식에서는 훨씬 빠랐음. 2회차 실행은 ~10초로 캐시 히트 시 정상.

## 원인 분석

### 핵심 병목: SQLite 트랜잭션 미사용

`session-db.js`의 `sync()` 메서드에서 **1,472개 JSONL 파일**(443MB)을 처리하면서, 각 세션마다 개별 INSERT/DELETE를 수행하고 있음. **BEGIN/COMMIT 트랜잭션 래핑이 없음.**

SQLite는 트랜잭션 없이 개별 INSERT를 실행하면, **매 INSERT마다 디스크 fsync**가 발생함 (autocommit). 이것이 단일 트랜잭션 대비 **수십~수백 배 느린** 원인.

| 항목 | 현재 (autocommit) | 개선 후 (단일 트랜잭션) |
|------|-------------------|----------------------|
| 디스크 fsync | INSERT마다 1회 | 전체 sync에 1회 |
| 1,472 세션 처리 | 수천 회 fsync | 1회 fsync |
| 예상 시간 | 1분+ | 수초 |

### 세부 병목 지점

1. **`_upsertSession()`** — 세션당 INSERT OR REPLACE 1회 (autocommit)
2. **`_upsertMessages()`** — 세션당 DELETE 1회 + 메시지 N개 INSERT (각각 autocommit)
3. **`_upsertPlanContent()`** — plan당 INSERT OR REPLACE 1회

첫 실행 시 1,472 세션 × (1 upsert + 1 delete + 평균 N messages) = **수천 개의 개별 트랜잭션**.

### 기존 방식과의 차이

`.build-cache.json` 방식: 메모리에서 JSON 객체를 구성 → 한 번에 파일 쓰기 (fsync 1회)
SQLite 방식: 매 row마다 디스크 쓰기 (fsync 수천 회)

### 변경 내용

`sync()` 메서드 시작/끝에 트랜잭션 래핑 추가:

```js
sync(options = {}) {
  // ... 기존 코드 ...

  this.db.exec("BEGIN");       // ← 추가
  try {
    // ── Claude 세션 ── (기존 로직 그대로)
    // ── Plans ── (기존 로직 그대로)
    // ── Codex 세션 ── (기존 로직 그대로)

    this.db.exec("COMMIT");    // ← 추가
  } catch (err) {
    this.db.exec("ROLLBACK");  // ← 추가
    throw err;
  }

  return { ... };
}
```

이것만으로 수십 배 성능 개선이 예상됨. 추가 최적화는 불필요.

## 2회차 개선: 캐시 히트 경로 최적화 (2026-03-09)

### 문제

트랜잭션 래핑 적용 후 첫 실행은 4.5초로 해결됐지만, **2회차 실행(거의 변경 없음)도 7~10초** 소요. 캐시 히트인데도 느린 이유 분석.

### 원인 분석

2회차 실행에서 변경이 0건이어도 수행되는 작업:

| 단계 | 작업 | 호출 수 (1292개 항목 기준) |
|------|------|--------------------------|
| sync() — stat | 모든 .jsonl/.md 파일 `statSync()` | ~1292회 |
| sync() — readdir | 프로젝트 디렉토리 + Codex year/month/day 순회 | ~100회 |
| **build.js — getMessages** | **세션 1292개 × SELECT 1회씩** | **~1292회 DB 쿼리** |
| build.js — JSON.stringify | allMeta + sessionsData 전체 직렬화 | 2회 (대용량) |
| build.js — HTML 쓰기 | 매번 index.html 재생성 | 1회 |

**핵심**: `build.js`는 sync()가 변경 0건을 반환해도 항상 전체 데이터를 DB에서 다시 읽고 HTML을 재생성.

**추가 발견 (계획에 없던 버그)**: Codex cacheKey 불일치

`_syncCodexDir`에서 cacheKey를 파일명 전체 basename으로 생성했으나, DB session_id는 `processCodexSession`이 `payload.id`(UUID만)를 사용하여 `"codex:UUID"` 형식으로 저장. 실제 파일명은 `rollout-2026-02-13T14-07-27-UUID.jsonl` 형태여서 cacheKey(`"codex:rollout-...-UUID"`)와 DB session_id(`"codex:UUID"`)가 불일치 → **매 실행마다 376개 Codex 세션 캐시 미스 발생**.

### 개선 내용

**변경 1: `shared/session-db.js` — Codex cacheKey UUID 추출 수정**

파일명에서 정규식으로 UUID 패턴을 추출하여 cacheKey를 DB session_id와 일치시킴:

```js
// 변경 전
const cacheKey = "codex:" + path.basename(file, ".jsonl");

// 변경 후
const basename = path.basename(file, ".jsonl");
const uuidMatch = basename.match(/([0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12})$/i);
const cacheKey = uuidMatch ? "codex:" + uuidMatch[1] : "codex:" + basename;
```

**변경 2: `my-session-dashboard/build.js` — early return 추가**

`sync()` 반환값에서 신규 항목이 0이고 출력 HTML이 이미 존재하면 rebuild 생략:

```js
const totalNew = stats.claudeNew + stats.planNew + stats.codexNew;
if (totalNew === 0 && fs.existsSync(htmlDest)) {
  db.close();
  console.log("✅ 변경 없음 — 기존 HTML 유지");
  return;
}
```

### 의사결정 근거

- **Codex cacheKey 수정 우선**: early return 단독으로는 Codex 미스 376개가 남아 totalNew > 0이 되어 early return이 발동하지 않음. 두 수정이 함께 작동해야 최적 효과.
- **변경 2(sync() 디렉토리 mtime 선필터)는 보류**: 변경 1+2 조합으로 2회차가 이미 0.76초로 목표치(2~3초 이하) 달성. stat 1292회는 OS 캐시로 ~0.2초 수준이므로 추가 최적화 불필요.

### 실측 결과

계획 실행 후 실측한 성능 데이터:

| 조건 | 항목 수 | 소요 시간 |
|------|---------|----------|
| 첫 실행 (DB 없음, 전체 신규) | Claude 659 + 플랜 257 + Codex 376 = 1292개 | **4.5초** |
| 2회차 실행 (캐시 히트) | 대부분 캐시 | **2.1초** |

예상(수초)보다도 훨씬 빠른 결과. WAL 모드(`PRAGMA journal_mode=WAL`)가 이미 적용되어 있었고, 트랜잭션 래핑이 추가로 fsync를 1회로 줄인 것이 핵심.

### 의사결정 근거

- **단일 트랜잭션 래핑만 적용**: `BEGIN` ~ `COMMIT` 3줄 추가로 수십 배 개선이 가능하므로, 추가 최적화(prepared statement 재사용, batch insert 등)는 불필요 판단.
- **에러 시 ROLLBACK 포함**: 부분 동기화로 DB가 오염되는 것을 방지하기 위해 try/catch 구조로 래핑.
- **`_buildSlugMap()` 위치 유지**: Plan 처리 중 DB 조회가 필요한데, 트랜잭션 내부에서 READ는 허용되므로 위치 변경 없이 그대로 유지.
