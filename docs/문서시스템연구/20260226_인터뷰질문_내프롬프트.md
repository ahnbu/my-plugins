# 인터뷰 질문


  Q1. 가장 아픈 곳은 어디인가요?

  현재 문서 관리에서 실제로 불편한 점을 우선순위로 말씀해주세요:

- (a) changelog 누락 (1단계에서 해결 예정)
- (b) 세션 간 컨텍스트 단절 (이전 세션 작업을 새 세션에서 모름)
- (c) 문서 간 역할 혼동 (어디에 뭘 적어야 할지 모호)
- (d) 계획 문서(~/.claude/plans/) 찾기 어려움
- (e) 기타

  Q2. "자기개선 루프"에 관심이 있나요?

  조아영 사례처럼 wrap에서 수집한 피드백을 다음 세션 시작 시 리마인드하는 기능 — 이게 현재 워크플로우에 가치가 있을까요? 아니면 auto memory + handoff로 충분한가요?

  Q3. context.md(프로젝트 고유 지식) 개념이 필요한가요?

  현재 CLAUDE.md에 시스템 규칙과 프로젝트 고유 지식이 섞여 있는데, 이를 분리할 필요성을 느끼시나요? 아니면 현재 CLAUDE.md 하나로 충분한가요?

3. 인터뷰

  분석이 넓어졌으니 질문도 조정합니다.

  Q1. 세션 간 단절이 실제로 문제인가요?

  handoff를 작성하고 있지만, 다음 세션에서 이전 handoff를 실제로 참조하는 빈도는 어느 정도인가요?

- (a) 거의 항상 읽고 시작한다
- (b) 가끔 읽는다 (특히 복잡한 작업 이어갈 때)
- (c) 거의 안 읽는다 — 새 세션에서 처음부터 설명하는 편
- (d) 기타

  Q2. wrap에서 더 뽑아내고 싶은 것이 있나요?

  현재 wrap은 handoff + commit입니다. team-attention처럼 추가 분석을 원한다면 어떤 것이 가장 가치 있을까요?

- (a) 자동화 기회 추천 (automation-scout)
- (b) 교훈/실수 추출 (learning-extractor)
- (c) 현재 handoff + commit으로 충분
- (d) 기타

  Q3. "자기개선 루프"의 형태는?

  조아영 사례의 /morning 리마인드처럼 이전 세션 피드백을 다음 세션에 자동 연결하는 기능에 관심이 있나요?

- (a) SessionStart 훅으로 자동 안내 (미반영 피드백 리스트 표시)
- (b) /morning 같은 별도 커맨드
- (c) auto memory가 이미 이 역할을 해서 불필요
- (d) 관심 있지만 지금 우선순위는 아님

- (c) 거의 안 읽음

- (b) 교훈/실수 추출 (team-attention learning-extractor)
- (c) 현재 handoff + commit으로 충분
- (d) 기타

  Q3. "사람용 문서" vs "기계용 상태 파일" 분리에 관심 있나요?

  bkit처럼 handoff(사람이 읽는 인수인계)와 별도로 .status.json(시스템이 자동 참조하는 상태)을 두는 방식. 세션 시작 시 hook이 상태 파일을 읽어 자동으로 컨텍스트를 주입할 수 있습니다.

- (a) 관심 있음 — 자동 컨텍스트 주입이 가치 있다
- (b) handoff 하나로 충분 — 복잡성 증가 대비 효과 불확실
- (c) 기타

  Q4. 자기개선 루프의 형태는?

- (a) SessionStart 훅으로 미반영 피드백 자동 표시
- (b) /morning 같은 별도 커맨드
- (c) auto memory가 충분
- (d) 지금 우선순위 아님
