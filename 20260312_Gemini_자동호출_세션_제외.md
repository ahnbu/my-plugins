## 발단: 사용자 요청

Gemini 세션 DB 통합 5단계(UI)까지 완료 후, Codex가 Gemini CLI를 쉘로 호출한 세션이 사용자 직접 대화와 구분 없이 동일 처리되는 문제 발견.

사용자 결정: **자동 호출 세션은 완전히 제외** (파싱/DB/대시보드에서 스킵)

## 작업 상세내역

### 문제 원인 분석

Codex가 `gemini` CLI를 쉘 명령으로 호출하면 Gemini 세션 JSON 파일이 생성된다. 이 파일의 구조는 사용자 직접 대화 세션과 동일하다:

- `sessionId`, `projectHash`, `startTime`, `lastUpdated`, `messages`, `kind` 필드만 존재
- 호출 출처(Codex인지 사용자인지) 식별 필드 없음
- `CODEX_THREAD_ID` 환경변수가 Gemini child process에 상속되지 않음 (실험 확인)

### 관찰된 데이터

- 자동 호출 세션: 지속시간 8~17초
- 사용자 직접 대화 세션: 지속시간 95초+
- 격차가 충분하여 60초 임계값으로 구분 가능

### bd4f4168 미표시 이슈 — 해소

별도 세션에서 DB 조회 결과 `gemini:bd4f4168-...`이 이미 존재 확인. build 미실행 또는 UI 필터 상태가 원인. 추가 진단 불필요.

## 구현 계획

### Step 1: 지속시간 60초 미만 세션 제외

**수정 파일**: `shared/session-db.js` — `_syncGeminiDir()` (L336~338)

파싱 후 `metadata.timestamp`와 `metadata.lastTimestamp` 차이를 계산하여 60초 미만이면 스킵:

```js
const result = processGeminiSession(absPath, projectRoot);
if (!result || result.metadata.userEntryCount === 0) continue;

// 자동 호출 세션 제외: 60초 미만
const durationSec = (new Date(result.metadata.lastTimestamp) - new Date(result.metadata.timestamp)) / 1000;
if (durationSec < 60) continue;

this._upsertSession(result.metadata, mtime);
```

### Step 2: `(codex)` 태그 감지 제외

동일 위치에서 첫 메시지에 `(codex)` 태그가 포함되면 지속시간 관계없이 제외:

```js
// (codex) 태그 감지
if (result.metadata.firstMessage && /\(codex\)/i.test(result.metadata.firstMessage)) continue;
```

### Step 3: 기존 자동 호출 세션 DB 정리

이미 DB에 들어간 60초 미만 세션 + `(codex)` 태그 세션 삭제. build.js 실행 시 자동 정리되도록 sync 전 cleanup 메서드 추가 또는 build 시 `--clean-gemini` 플래그로 Gemini 세션 전체 재동기화.

**간단한 접근**: Gemini 세션만 DELETE 후 재sync:

```js
// build.js에서 호출
db.db.exec("DELETE FROM messages WHERE session_id LIKE 'gemini:%'");
db.db.exec("DELETE FROM sessions WHERE type = 'gemini'");
```

→ 이후 `_syncGeminiDir`가 필터링 적용하여 재삽입. 1회성 정리.
