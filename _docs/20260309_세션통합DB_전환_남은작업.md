## 작업 순서 (쉬운 것 → 어려운 것)

### Step 1: session-loader.js — DB에서 본문까지 읽기 (쉬움 ⭐)

**현재**: DB에서 file_path만 조회 → readJsonl()로 JSONL 파싱
**변경**: `SessionDB.getEvents(sessionId)`로 정규화된 이벤트 직접 조회, DB에 없을 때만 JSONL 폴백

- 파일: `my-session-wrap/lib/session/session-loader.js`
- 의존: `shared/session-db.js`의 `getEvents()` 이미 구현됨
- 위험: 낮음 — DFS 폴백 유지하므로 DB 누락 시에도 안전
- 검증: `node my-session-wrap/tests/session-tools.test.js` (21개 테스트)

### Step 2: session-normalizer.js — DB events 경로 추가 (쉬움 ⭐)

**현재**: `normalizeSessionBundle(bundle)` — bundle.mainEntries (JSONL raw)를 normalizeEntries()로 변환
**변경**: DB events가 이미 정규화 완료이므로, DB에서 가져온 경우 normalizeEntries() 스킵

- 파일: `my-session-wrap/lib/session/session-normalizer.js`
- 핵심: bundle에 `fromDb: true` 플래그가 있으면 이미 정규화된 events를 그대로 반환
- 위험: 낮음 — 기존 JSONL 경로 그대로 유지, 새 경로 추가만
- 검증: 기존 테스트 통과 + timeline/transcript 출력 비교

### Step 3: session-timeline — DB 데이터 소스 전환 (중간 ⭐⭐)

**현재**: `loadSessionBundle()` → readJsonl() → normalizeSessionBundle() → buildTimeline()
**변경**: DB에서 events 조회 → buildTimeline() (로드+정규화 단계 대체)

- 파일: `my-session-wrap/skills/session-timeline/session_timeline.js`
- 핵심: `loadSessionBundle()` 내부에서 DB 우선 조회 분기 추가
- 의존: Step 1~2 완료 필요
- 검증: 실제 세션 ID로 timeline 실행, 기존 출력과 비교

### Step 4: session-transcript — DB 데이터 소스 전환 (중간 ⭐⭐)

**현재**: session-timeline과 동일한 loadSessionBundle() 파이프라인
**변경**: Step 3과 동일 패턴 적용

- 파일: `my-session-wrap/skills/session-transcript/session_transcript.js`
- 핵심: `loadSessionBundle()` 공유 함수라면 Step 3에서 이미 전환 완료 가능
- 검증: 실제 세션 ID로 transcript 실행, 기존 출력과 비교

### Step 5: Stop 훅 — 현재 세션 즉시 DB upsert (중간 ⭐⭐)

**이슈 검토 결론**: Hybrid C 방식 — Stop 훅에서 현재 세션 syncSingleSession() + 조회 시 증분 sync 보정

- 파일: `my-session-wrap/hooks/hooks.json` (Stop 이벤트 추가) + 새 훅 스크립트
- 핵심: Stop 이벤트의 stdin에서 sessionId 추출 → `SessionDB.syncSingleSession(sessionId)`
- 주의: mtime 비교로 변경 없으면 즉시 return (수십ms), timeout 3000ms 설정
- 위험: 중간 — 훅 실패 시 DB 누락이지만, 조회 시 sync 보정으로 안전
- 검증: Stop 훅 실행 후 DB에 세션 레코드 존재 확인

### Step 6 (보류): history-insight — DB 기반 전환 (어려움 ⭐⭐⭐)

- 현재 bash/jq 파이프라인 → JS 스크립트로 재작성 필요

- **이번 범위 제외. 추후 검토 후 반영.**

- **이번 범위 제외. 추후 검토 후 반영.**

## 보충: 대화 중 검토 내용

### 의사결정 근거

**왜 이 순서인가 (쉬운 것 → 어려운 것)**
- Step 1~2는 기존 인프라(shared/session-db.js)가 이미 준비된 상태에서 소비자만 연결하면 됨. 위험도 최저.
- Step 3~4는 timeline/transcript 파이프라인이 동일 패턴이라 Step 3 완료 시 Step 4가 거의 자동으로 따라옴.
- Step 5는 새 훅 스크립트 작성 + hooks.json 수정이 필요해 상대적으로 복잡.

**Stop 훅 채택 이유 (이슈 검토 2-3 결론)**

| 기준 | A. Pull-only | B. Stop 훅만 | C. Hybrid (채택) |
|------|:---:|:---:|:---:|
| `/ss` 빌드 속도 | ⚠️ 첫 호출 느림 | ████░░ 대부분 DB 존재 | ✅ ██████ 미반영분만 증분 |
| 외부 세션 반영 | ✅ 전체 스캔 보장 | ❌ 훅 미실행 시 누락 | ✅ 조회 시 보정 |
| 훅 실패 영향 | ✅ 해당 없음 | ⚠️ DB 누락 | ✅ 조회 시 보정 |
| 구현 복잡도 | ✅ 낮음 | 보통 | ⚠️ 두 경로 |

→ Stop 훅이 단건 upsert로 빠르고, 실패해도 조회 시 sync 보정으로 안전.

**fs.watch 데몬 미채택 이유**
- 다중 세션 중복 spawn, 프로세스 고아화 위험
- Windows fs.watch 안정성 이슈 (중복 이벤트, 락 충돌)
- Stop 훅이 동일 효과를 더 단순하게 달성
