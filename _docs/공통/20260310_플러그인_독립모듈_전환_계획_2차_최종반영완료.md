- Plugin 시스템 존재 목적 분석이 정확 ("공유·배포 전용 설계")
- 시나리오별 비교표(🅰️🅱️🅲️)가 명확 — 본인 전용 도구에 Plugin이 과잉임을 증명

- **결정**: 🅲️ Option C (Plugin 미설치 운용)
- **근거**:
  - 전환 비용 최소 (plugin uninstall + settings.json 수정)이면서 핵심 문제(이중 경로) 해결
  - Plugin 소스 구조 그대로 유지 → 배포 필요 시 즉시 가능
  - 사용자 직관 ②("나는 my-claude-plugin 설치하지 않고 사용한다")가 정확
  - Option B의 중간 단계 — 불편하면 언제든 B로 진행 가능
- **트레이드오프**: Plugin 소스 구조(.claude-plugin/, hooks.json)는 사용하지 않는 코드로 남음. 배포 시 테스트 필요.

**Option D(Junction) 배제 이유:**
- Windows junction은 관리자 권한 이슈, MSYS2/Git Bash 호환성 문제 가능
- Claude Code의 plugin update가 junction을 덮어쓸 위험
- 문제를 은폐할 뿐 해결하지 않음

**Option A(현행) 배제 이유:**
- post-commit 자동화가 있어도 "3중 경로 중 어디가 정본?"이라는 인지 부하 해소 불가
- shared/ 참조가 ①② 어디서 실행되느냐에 따라 다른 DB를 읽는 문제 지속

## 주의사항

1. **settings.json hooks와 plugin hooks.json은 동시 실행됨** — plugin uninstall 먼저, settings.json 나중에
2. **`${CLAUDE_PLUGIN_ROOT}` 변수 소멸** — plugin 미설치 시 이 변수가 주입되지 않음. hook 스크립트 내 참조 부분을 `__dirname` 등으로 대체 필요
3. **세션 재시작 필수** — settings.json hooks 변경은 현재 세션에 반영 안 됨. 새 세션에서 검증

### 의사결정: inject-plugin-guidelines.js 처리 방향

- **배경**: 이 hook은 plugin 설치 체제에서 배포 3단계(git push → marketplace update → plugin update) 절차를 세션 시작마다 강제 주입하기 위해 만들어진 것. git push가 필요했던 이유도 "배포해야 사용할 수 있기 때문"이었음.
- **미설치 모드에서**: push는 post-commit hook(`auto-push-update.js`)이 이미 자동 처리. 배포 절차 주입 자체가 불필요.
- **결정**: 수정이 아니라 제거. `.bak`으로 보존하여 타인 배포 시 복원 가능하도록 유지.

### 의사결정: git-hooks 처리 방향

- **배경**: `git-hooks/auto-push-update.js`는 plugin 배포 자동화 전용. 커밋 후 git push → marketplace update → plugin update 체인을 담당. `git-hooks/post-commit`은 이를 호출하는 shell 래퍼 (`core.hooksPath = git-hooks`로 활성화).
- **미설치 모드에서**: push가 배포 체인의 시작점이었으나, 배포 자체가 없으므로 자동 push도 불필요. git push는 필요 시 수동으로 실행.
- **결정**: `auto-push-update.js`, `post-commit` 모두 `.bak`으로 보존. 배포 복원 시 함께 복원.
