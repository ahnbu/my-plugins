## 2. 사례별 상세 분석

### A. team-attention session-wrap

| 항목                     | 내용                                                                                                                                             |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| **문제**           | 세션 종료 시 "CLAUDE.md에 적어야 하나?", "다음에 뭐해야 하더라?"                                                                                 |
| **메커니즘**       | `/wrap`: Phase 1 — 4에이전트 병렬(doc-updater, automation-scout, learning-extractor, followup-suggester) → Phase 2 — duplicate-checker 검증 → 사용자 선택 |
| **추가 커맨드**    | `/history-insight`: 세션 기록 분석 (현재 프로젝트 또는 전체 세션 검색, 테마·결정사항·반복 주제 추출) · `/session-analyzer`: 사후 세션 검증 (SKILL.md 규격 대비 에이전트·훅·도구 실행 준수 보고서) |
| **문서 모델**      | CLAUDE.md(시스템 규칙) + context.md(프로젝트 고유 지식)                                                                                          |
| **세션 간 연속성** | CLAUDE.md/context.md 업데이트로 영구 축적 + `/history-insight`로 과거 세션 크로스 탐색                                                           |
| **교훈 축적**      | learning-extractor(세션 내 TIL 추출) + `/history-insight`(크로스세션 패턴·테마 분석)                                                              |
| **자동화 발견**    | automation-scout가 반복 패턴 감지 → skill/command/agent 제안                                                                                    |
| **품질 검증**      | duplicate-checker(중복 검증) + `/session-analyzer`(SKILL.md 규격 대비 실행 준수 검증)                                                             |
| **강점**           | 세션 분석의 다차원성 — 문서·자동화·교훈·후속작업 포괄 + 사후 검증·크로스세션 분석까지 커버                                                    |
| **약점**           | 5에이전트 = 높은 토큰 비용 + 복잡성. [짧은 세션에도 풀 파이프라인]()                                                                                |

### B. bkit (Vibecoding Kit)

| 항목                     | 내용                                                                             |
| ------------------------ | -------------------------------------------------------------------------------- |
| **문제**           | AI 코딩에서 설계↔구현 불일치, 목표 표류, 문서화 누락                            |
| **메커니즘**       | PDCA 사이클 (`/pdca plan → design → do → analyze → iterate → report`)     |
| **문서 모델**      | 4-폴더 구조:`01-plan/` → `02-design/` → `03-analysis/` → `04-report/` |
| **상태 관리**      | `.pdca-status.json`(기능별 진행 상태) + `.bkit-memory.json`(세션 메타)       |
| **에이전트**       | 16개 — gap-detector, pdca-iterator, report-generator, cto-lead 등               |
| **Hook 시스템**    | 10개 이벤트, 45개 스크립트                                                       |
| **자동 반복**      | Check→Act 루프: Match Rate < 90%이면 최대 5회 자동 수정                         |
| **세션 간 연속성** | `.pdca-status.json` history 배열 + phase/matchRate 영속 저장                   |
| **교훈 축적**      | 완료 보고서 Lessons Learned 섹션 + archive 시스템 (월별)                         |
| **프로젝트 레벨**  | Starter/Dynamic/Enterprise 자동 감지                                             |
| **강점**           | 체계적 문서 구조, [기계용/사람용 분리](), 자동 반복 개선                            |
| **약점**           | 매우 높은 복잡도 (27 skills, 16 agents, 241 functions). [코딩 프로젝트 특화]()      |

## 5. 기존 계획(handoff_02) 개선 검토

### 5.1 현재 계획의 강점

- 1단계(changelog 안내)는 80/20 원칙에 충실
- team-attention의 과잉 복잡성을 배제한 판단이 적절

### 5.2 bkit에서 배울 점

| bkit 요소               | 인사이트                                                 | 우리 환경 적용 가능성                               |
| ----------------------- | -------------------------------------------------------- | --------------------------------------------------- |
| `.pdca-status.json`   | JSON 상태 파일로 세션 간 연속성 보장. 시스템이 자동 참조 | handoff(사람용)과 별도로 기계용 상태 파일 도입 검토 |
| 4-폴더 문서 구조        | 단계별 산출물 명확                                       | PDCA 전체는 과잉이나, 검토 문서 체계화에 참고 가능  |
| Hook 10개 이벤트        | 다양한 시점에 개입                                       | Stop hook으로 상태 자동 저장은 가치 있음            |
| Match Rate + 자동 반복  | 설계↔구현 갭 수치화                                     | 코딩 프로젝트 아닌 환경에서는 해당 없음             |
| 아카이브 시스템         | 완료 문서 월별 정리                                      | handoff 누적 시 유사한 정리 필요                    |
| 프로젝트 레벨 자동 감지 | 복잡도별 차별화                                          | 흥미롭지만 현재 불필요                              |

### 5.3 전체 사례 기반 개선 방향

| 우선순위     | 항목                             | 근거 사례                                  | 복잡도             |
| ------------ | -------------------------------- | ------------------------------------------ | ------------------ |
| **P0** | changelog 안내 (기존 1단계)      | handoff_02 기존 계획                       | 낮음               |
| **P1** | 세션 시작 시 handoff 재참조 안내 | 마누스 "결정 전 재참조", bkit SessionStart | 낮음               |
| **P2** | 피드백→리마인드 경로            | 조아영 루프                                | 중간               |
| **P3** | 문서 역할 지도 정립              | 5개 사례 비교 결과                         | 설계 작업          |
| **P4** | 기계용 상태 파일 도입 검토       | bkit `.pdca-status.json`                 | 중간               |
| **P5** | automation-scout 경량 버전       | team-attention                             | 높음 (별도 커맨드) |

## 6. 인터뷰 질문

### Q1. 세션 간 단절이 실제로 문제인가요?

다음 세션에서 이전 handoff를 참조하는 빈도는?

- (a) 거의 항상
- (b) 복잡한 작업 이어갈 때만
- (c) 거의 안 읽음
- (d) 기타

### Q2. wrap에서 더 뽑아내고 싶은 것은?

- (a) 자동화 기회 추천 (team-attention automation-scout)
- (b) 교훈/실수 추출 (team-attention learning-extractor)
- (c) 현재 handoff + commit으로 충분
- (d) 기타

### Q3. "사람용 문서" vs "기계용 상태 파일" 분리에 관심 있나요?

bkit처럼 handoff(사람이 읽는 인수인계)와 별도로 `.status.json`(시스템이 자동 참조하는 상태)을 두는 방식. 세션 시작 시 hook이 상태 파일을 읽어 자동으로 컨텍스트 주입 가능.

- (a) 관심 있음 — 자동 컨텍스트 주입이 가치 있다
- (b) handoff 하나로 충분 — 복잡성 증가 대비 효과 불확실
- (c) 기타

### Q4. 자기개선 루프의 형태는?

- (a) SessionStart 훅으로 미반영 피드백 자동 표시
- (b) /morning 같은 별도 커맨드
- (c) auto memory가 충분
- (d) 지금 우선순위 아님
